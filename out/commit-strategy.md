### 4.2 BEP (Break-Even Point) 계산식 및 결과

**공식**:  
`BEP(개월) = 선납비용 / 월간 RI 절감액`  
`선납비용(1y All Upfront) = OD 월비용 × 12 × (1 − 할인율)`  
`월간 RI 절감액 = OD 월비용 × 할인율`  
**=> BEP(1y All Upfront) = 12 × (1 − d) / d**

| 구분 | Conservative | Base | Optimistic |
|------|-------------:|-----:|-----------:|
| RI 적용분 OD 월비용(KRW) | 12,566,248 | 15,707,811 | 18,849,373 |
| **RI 1y 선납(All Upfront, KRW)** | 105,556,487 | 131,945,608 | 158,334,730 |
| **BEP(1y) 개월** | **28.0** | **28.0** | **28.0** |
| RI 3y 선납(All Upfront, KRW) | 226,192,471 | 282,740,589 | 339,288,707 |
| **BEP(3y) 개월** | **36.0** | **36.0** | **36.0** |

> **BEP가 시나리오별 동일한 이유**: All Upfront 공식상 BEP는 RI 적용 비율과 무관하게  
> **할인율 d로만 결정됨** (12×(1−d)/d). RI 비율은 절감 총액과 선납 총액에 동시 영향.  
> Partial Upfront/No Upfront 채택 시 BEP는 단축됨(매트릭스 §4 참조).

## 5. 시나리오별 상세 분석

### 5.1 Conservative

- **적합**: HBT FinOps 도입 1년차 초기, 1개월 샘플밖에 없는 현 시점에 가장 적합  
- RI 1y No/Partial Upfront 우선 검토 권고 (선납 부담 완화로 BEP < 12개월 가능)  
- Spot 미사용으로 인터럽트 리스크 0  
- 월 4.56M KRW 절감, 연 54.7M KRW 회수 — 보수적이나 안전

### 5.2 Base

- **적합**: 3~6개월 추가 데이터 수집 후, 트래픽 패턴 확인 시 전환  
- 월 8.48M KRW 절감으로 Conservative 대비 +86% 효과  
- Spot 10%는 무상태 변동 워크로드(vm-spike-azure·i-spike-001·gce-spike-1)에 한정 적용 권고  
- SP 20%로 인스턴스 패밀리 변경 유연성 확보

### 5.3 Optimistic

- **적합**: 1개월 샘플 단계에서는 비권고. 1년 이상 안정 운영 검증 후 검토  
- 월 10.92M KRW 절감(연 130.0M)으로 최대 효과지만 매몰비 위험  
- 선납 158M KRW(1y) 또는 339M KRW(3y) 자금 사전 확보 필요  
- 서비스 폐기·축소 가능성 검토 + 재무팀 사전 협의 필수

## 6. 권고 시나리오

### 6.1 권고: **Conservative 즉시 적용 + Base 단계적 전환**

**근거**:

1. **데이터 한계**: 1개월 샘플로 Optimistic의 90% 약정 커버리지 결정 불가  
2. **베이스라인 변동성**: Right-sizing 적용 자체로 베이스라인이 -23% 이동 예정  
   → 약정 베이스를 잘못 잡을 위험 큼  
3. **Conservative ROI**: BEP 28개월(1y All Upfront)로 1년 약정 회수 불가하므로  
   **1y Partial Upfront 또는 No Upfront 우선** 권고 (선납 축소로 회수 기간 단축)  
4. **Base 전환 조건**: 3개월 추가 hourly 데이터 + 분기 트래픽 패턴 확인 후 전환

### 6.2 비권고

- **3y All Upfront 즉시 도입 비권고**: BEP 36개월로 약정 기간 100% 시 회수 → 마진 없음  
- **Optimistic 즉시 도입 비권고**: 데이터 부족 + 매몰비 리스크
